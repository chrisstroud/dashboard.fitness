## User Story

**As a** dashboard user
**I want to** delete a habit stack without losing its protocols
**So that** I can reorganize my routine structure safely

---

## Acceptance Criteria

- [ ] **AC1:** `DELETE /api/protocols/groups/{id}` reassigns all protocols in the deleted group to the first remaining group in the section (sorted by position)
- [ ] **AC2:** Returns `400` with error message if the group is the last one in its section
- [ ] **AC3:** Returns `404` if the group doesn't exist or belongs to another user
- [ ] **AC4:** Reassigned protocols get new sequential positions after the target group's existing protocols
- [ ] **AC5:** Returns `204` on successful deletion
- [ ] **AC6:** Daily instance is refreshed after deletion (`_commit_and_refresh()`)

## Implementation Guidance

### Approach

Replace the existing `delete_group` handler with one that: (1) verifies the group isn't the last in its section, (2) moves all protocols to the first remaining group, (3) deletes the now-empty group.

### Layer Sequence

1. API layer only — modify existing route handler in `api/routes/protocols.py`

### Step-by-Step

1. Replace the existing `delete_group` function (line 152-157) with:

```python
@protocols_bp.route("/groups/<group_id>", methods=["DELETE"])
def delete_group(group_id: str):
    """Delete a group (habit stack), reassigning its protocols to the first remaining group."""
    group = db.get_or_404(ProtocolGroup, group_id)

    # Verify ownership
    if not group.section or group.section.user_id != g.user_id:
        return jsonify({"error": "Not found"}), 404

    section = group.section
    remaining = [gr for gr in section.groups if gr.id != group.id]

    if not remaining:
        return jsonify({"error": "Cannot delete the last stack in a section"}), 400

    # Reassign protocols to the first remaining group (by position)
    target = sorted(remaining, key=lambda gr: gr.position)[0]
    max_pos = max((p.position for p in target.protocols), default=-1)
    for i, proto in enumerate(sorted(group.protocols, key=lambda p: p.position)):
        proto.group = target
        proto.position = max_pos + 1 + i

    db.session.delete(group)
    _commit_and_refresh()
    return "", 204
```

2. Test with curl:
```bash
# Delete a group that has protocols — verify they move to the remaining group
curl -X DELETE http://localhost:5001/api/protocols/groups/$GROUP_ID \
  -H "Authorization: Bearer $TOKEN"

# Try to delete the last group in a section — should get 400
curl -X DELETE http://localhost:5001/api/protocols/groups/$LAST_GROUP_ID \
  -H "Authorization: Bearer $TOKEN"
```

### Gotchas / Watch Out For

- **SQLAlchemy cascade:** The existing `ProtocolGroup` model likely has `cascade="all, delete-orphan"` on its protocols relationship. Reassigning `proto.group = target` before `db.session.delete(group)` should move the protocol out of the cascade scope. **Verify this works** — if SQLAlchemy still cascades, you may need to flush the session between reassignment and deletion: `db.session.flush()` then `db.session.delete(group)`.
- **Empty group deletion:** If the group has zero protocols, it should still be deletable (as long as it's not the last group).
- **Return value change:** The existing endpoint returns `jsonify({"deleted": True})`. Changing to `return "", 204` is a breaking change for any client expecting the JSON body. Since only the iOS app calls this and it's being rewritten, this is acceptable.

## Dev Notes

### Architecture Constraints
- Every section must have at least one group (stack) at all times
- Protocol reassignment preserves the protocols' relative order within the new group
- Must call `_commit_and_refresh()` to update today's daily tasks

### Patterns to Follow
- `db.get_or_404()` for initial lookup
- Ownership check via `group.section.user_id != g.user_id`
- `_commit_and_refresh()` for all mutations

### Previous Learnings
- Sync destructiveness: earlier in this project, `syncAll()` deleted local sections not found server-side. Deletions that cascade unexpectedly are a recurring risk. The protocol reassignment pattern prevents data loss.

## Testing Notes

### Test Scenarios
1. **Happy path:** Delete group with 3 protocols, section has 2 groups → protocols move to other group
2. **Last group guard:** Delete the only group in a section → returns 400
3. **Empty group:** Delete group with 0 protocols → deletes cleanly, returns 204
4. **Ownership:** Try to delete another user's group → returns 404

### Edge Cases
- Group with protocols where the target group also has protocols: positions should be max(target) + 1, +2, etc.
- Group with protocols where the target group is empty: positions start at 0
- Concurrent deletion: two groups deleted simultaneously — second delete may try to reassign to the first (now deleted). `db.get_or_404` on the group handles this.
