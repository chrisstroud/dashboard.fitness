## Technical Context

### Architecture Reference

SwiftUI `List` with `ForEach(...).onMove(perform:)` handles reorder natively when the view is in edit mode. Each level's `ForEach` gets its own `onMove` closure that receives `IndexSet` (source indices) and `Int` (destination index).

## Implementation Guidance

### Approach

Add `@State private var editMode: EditMode = .inactive` and bind it to the List via `.environment(\.editMode, $editMode)`. Add `EditButton()` in the toolbar. Add `.onMove` modifiers to the three `ForEach` loops created in Story 6.2. Each reorder handler sorts the affected items, applies `Array.move(fromOffsets:toOffset:)`, then renumbers positions.

### Layer Sequence

1. UI layer — modify `MasterTemplateEditor.swift`
2. Note: position sync to API is Story 7.2 — this story does local SwiftData writes only

### Step-by-Step

1. **Add edit mode state:**

```swift

```

2. **Bind edit mode to the List and add EditButton:**

```swift
List { ... }
    .environment(\.editMode, $editMode)
    .toolbar {
        ToolbarItem(placement: .primaryAction) {
            HStack {
                EditButton()  // ← toggles editMode

            }

```

Alternatively, keep the `+` Menu as primary and put `EditButton` as secondary:

```swift
.toolbar {
    ToolbarItem(placement: .topBarLeading) {
        EditButton()
    }
    ToolbarItem(placement: .primaryAction) {
        Menu { ... }  // existing "+" menu
    }
}
```

3. **Add `.onMove` to sections ForEach:**

Note: SwiftUI `List` `Section` views cannot themselves be reordered with `.onMove` — `onMove` works on `ForEach` rows within a section. To reorder sections, we need a different approach. Options:

**Option A: Use a flat List structure** where sections are rendered as styled rows (not `Section {}` containers). This allows `.onMove` on the outer ForEach but loses the native List section styling.

**Option B: Keep `Section` containers, add section reorder via context menu** (move up/move down buttons) instead of drag.

**Recommended: Option B** — keep native `Section` styling, use move up/down buttons on section headers for section reorder. Drag-and-drop applies to stacks and protocols only.

```swift
// In sectionHeader(_:):
Menu {
    // ... existing items
    if section.position > 0 {
        Button { moveSectionUp(section) } label: {
            Label("Move Up", systemImage: "arrow.up")
        }
    }
    if section.position < sections.count - 1 {
        Button { moveSectionDown(section) } label: {
            Label("Move Down", systemImage: "arrow.down")
        }
    }
}
```

4. **Add `.onMove` to stacks ForEach** (within each section):

```swift
ForEach(section.sortedGroups) { group in
    // ... stack header + protocol rows
}
.onMove { indices, destination in
    moveStacks(indices, destination, in: section)
}
```

5. **Add `.onMove` to protocols ForEach** (within each stack):

```swift
ForEach(protos) { proto in
    // ... NavigationLink + ProtocolRow
}
.onMove { indices, destination in
    moveProtocols(indices, destination, in: group)
}
.onDelete { offsets in
    for offset in offsets { modelContext.delete(protos[offset]) }
}
```

6. **Add reorder handler functions:**

```swift
// MARK: - Reorder Handlers

private func moveSectionUp(_ section: ProtocolSection) {
    let sorted = sections.sorted { $0.position < $1.position }
    guard let idx = sorted.firstIndex(where: { $0.id == section.id }), idx > 0 else { return }
    sorted[idx].position = idx - 1
    sorted[idx - 1].position = idx
    // Story 7.2 will add: SyncService.shared.syncPositions(sections: [sorted[idx], sorted[idx-1]])
}

private func moveSectionDown(_ section: ProtocolSection) {
    let sorted = sections.sorted { $0.position < $1.position }
    guard let idx = sorted.firstIndex(where: { $0.id == section.id }), idx < sorted.count - 1 else { return }
    sorted[idx].position = idx + 1
    sorted[idx + 1].position = idx
}

private func moveStacks(_ indices: IndexSet, _ destination: Int, in section: ProtocolSection) {
    var ordered = section.sortedGroups
    ordered.move(fromOffsets: indices, toOffset: destination)
    for (i, group) in ordered.enumerated() { group.position = i }
}

private func moveProtocols(_ indices: IndexSet, _ destination: Int, in group: ProtocolGroup) {
    var ordered = group.sortedProtocols
    ordered.move(fromOffsets: indices, toOffset: destination)
    for (i, proto) in ordered.enumerated() { proto.position = i }
}
```

7. **Prevent swipe-to-delete on sections and stacks:**

```swift
ForEach(section.sortedGroups) { group in ... }
    .onMove { ... }
    .deleteDisabled(true)  // ← stacks not swipe-deletable
```

Sections use native `Section {}` so they don't have swipe-delete by default.

### Code Patterns to Follow

```swift
// Standard SwiftUI onMove pattern:
ForEach(items) { item in
    Text(item.name)
}
.onMove { indices, destination in
    var items = items
    items.move(fromOffsets: indices, toOffset: destination)
    for (i, item) in items.enumerated() { item.position = i }
}
```

### Gotchas / Watch Out For

- **`@Query` sort order:** `@Query(sort: \ProtocolSection.position)` re-sorts automatically when positions change. This is correct behavior but means the list will animate to the new order after SwiftData writes.
- **`sortedGroups` returns a copy:** `section.sortedGroups` creates a new sorted array. The `move(fromOffsets:toOffset:)` mutates this copy, then the position writes go back to the SwiftData models. This works because each element in the sorted array is a reference to the same `@Model` object.
- **Edit mode + NavigationLink:** When edit mode is active, tapping a row enters reorder/delete mode instead of navigating. This is standard iOS behavior. `EditButton` toggles between `.active` and `.inactive`.
- **Section reorder via context menu, not drag:** This is a pragmatic trade-off. SwiftUI doesn't support dragging `Section` containers. The move up/down pattern is common in settings-style UIs.
