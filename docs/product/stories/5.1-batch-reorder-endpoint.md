## User Story

**As a** dashboard user
**I want to** have my reordered sections, stacks, and protocols saved to the server in one call
**So that** drag-and-drop changes persist without needing three separate API calls

---

## Acceptance Criteria

- [ ] **AC1:** `PATCH /api/protocols/reorder` accepts optional `sections`, `groups`, and `protocols` arrays in a single request
- [ ] **AC2:** Each array item contains `id` and `position`; groups optionally include `section_id`; protocols optionally include `group_id`
- [ ] **AC3:** Only items owned by the authenticated user are updated (ownership verified)
- [ ] **AC4:** Returns `204 No Content` on success
- [ ] **AC5:** Missing or empty arrays are silently skipped (partial updates work)
- [ ] **AC6:** Invalid IDs are silently skipped (no error for items not found)

## Implementation Guidance

### Approach

Add a single route handler on `protocols_bp` that accepts all three item types in one JSON body. Iterate each array, look up the item by ID, verify ownership, update `position` (and optionally `section_id`/`group_id` for cross-moves). Commit once. Call `_commit_and_refresh()` to update today's daily instance.

### Layer Sequence

1. API layer only — single route handler in `api/routes/protocols.py`

### Step-by-Step

1. Add the route handler after the existing `reorder_groups` function (around line 170):

```python
@protocols_bp.route("/reorder", methods=["PATCH"])
def batch_reorder():
    """Batch update positions for sections, groups, and/or protocols."""
    data = request.get_json()

    for item in data.get("sections", []):
        section = db.session.get(ProtocolSection, item["id"])
        if section and section.user_id == g.user_id:
            section.position = item["position"]

    for item in data.get("groups", []):
        group = db.session.get(ProtocolGroup, item["id"])
        if group and group.section and group.section.user_id == g.user_id:
            group.position = item["position"]
            if "section_id" in item:
                group.section_id = item["section_id"]

    for item in data.get("protocols", []):
        proto = db.session.get(Protocol, item["id"])
        if proto and proto.group and proto.group.section and proto.group.section.user_id == g.user_id:
            proto.position = item["position"]
            if "group_id" in item:
                proto.group_id = item["group_id"]

    _commit_and_refresh()
    return "", 204
```

2. Test with curl:
```bash
curl -X PATCH http://localhost:5001/api/protocols/reorder \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sections": [{"id": "uuid-1", "position": 1}, {"id": "uuid-2", "position": 0}]}'
```

### Code Patterns to Follow

```python
# Existing reorder pattern (sections/reorder, line 93):
@protocols_bp.route("/sections/reorder", methods=["PUT"])
def reorder_sections():
    data = request.get_json()
    for item in data.get("order", []):
        section = ProtocolSection.query.get(item["id"])
        if section:
            section.position = item["position"]
    _commit_and_refresh()
    return jsonify({"reordered": True})

# New endpoint follows same pattern but:
# 1. Uses PATCH instead of PUT (partial update semantics)
# 2. Handles all three levels
# 3. Verifies ownership via g.user_id
# 4. Returns 204 (no body needed)
```

### Gotchas / Watch Out For

- **Ownership traversal depth:** Protocols require `proto.group.section.user_id` — 3 levels deep. Each `db.session.get()` may return `None` if the ID is invalid. Guard all traversals with `and` checks.
- **`_commit_and_refresh()` calls `refresh_today()`:** This regenerates DailyTask positions from the master template. This is correct — after reordering protocols, the daily snapshot should reflect the new order.
- **Variable name collision:** The existing code uses `g` from Flask (`g.user_id`). The loop variable for groups in the architecture doc was also `g_item` — don't shadow Flask's `g`.

## Dev Notes

### Architecture Constraints
- Must call `_commit_and_refresh()` (not raw `db.session.commit()`) to keep daily instance in sync
- `section_id` and `group_id` fields in the payload support future cross-moves (Epic 8) — include them now

### Patterns to Follow
- Follow existing `_require_authentication()` guard — it runs automatically via `before_request`
- Follow existing `db.session.get()` pattern (not `query.get()` which is deprecated in SQLAlchemy 2.0)

### Previous Learnings
- The existing `reorder_sections` and `reorder_groups` endpoints don't verify ownership. This new endpoint should verify ownership to be correct.

## Testing Notes

### Test Scenarios
1. **Happy path:** Send all three arrays with valid IDs — verify positions updated in DB
2. **Partial update:** Send only `sections` array — verify groups and protocols untouched
3. **Empty body:** Send `{}` — returns 204, nothing changes
4. **Invalid IDs:** Include a nonexistent UUID — silently skipped, valid items still processed
5. **Cross-user:** Include another user's section ID — silently skipped

### Edge Cases
- All arrays empty: `{"sections": [], "groups": [], "protocols": []}` → 204, no-op
- Duplicate IDs in array: last one wins (standard dict-less iteration)
- `group_id` pointing to a group in a different user's section: ownership check on the protocol's current section prevents unauthorized access, but the new `group_id` target isn't validated — acceptable since the protocol already belongs to the user
